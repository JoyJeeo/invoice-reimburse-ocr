# 发票批量识别与报销Excel导出工具——需求手册（最终版）


## 一、项目概述

### 1.1 项目背景

在企业日常财务工作中，发票信息的录入是一项高频且重复性高的任务。传统方式依赖人工逐张识别发票信息（如发票代码、号码、金额、日期等），再手动录入Excel或财务系统，存在三大痛点：

- **效率低下**：单张发票录入需1-2分钟，月均处理数百张发票耗费大量人力；
- **易出错**：数字、日期等关键字段人工录入错误率约2%-3%；
- **难以追溯**：纸质发票易丢失，电子发票需人工分类存储。

自动化发票识别方案可将单张发票处理时间从分钟级降至秒级，准确率提升至95%以上。

### 1.2 项目目标

开发一款轻量级桌面工具，能够：
- 批量读取指定文件夹下的所有图片格式发票；
- 自动识别并提取发票中的关键报销信息；
- 将识别结果整理为结构化的Excel文件，输出到指定目录。

### 1.3 目标用户

- 企业财务人员、会计人员；
- 需要处理大量发票报销的中小企业员工；
- 审计、税务相关从业人员。


## 二、功能需求

### 2.1 功能模块总览

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 输入模块 | 读取指定文件夹下所有图片 | 高 |
| 图像预处理 | 图片质量优化 | 中 |
| OCR识别 | 发票文字识别与字段提取 | 高 |
| 数据解析 | 关键字段结构化提取 | 高 |
| 数据校验 | 识别结果校验与纠错 | 中 |
| Excel导出 | 生成结构化Excel报表 | 高 |
| 日志与异常处理 | 记录处理过程和错误 | 中 |

### 2.2 详细功能描述

#### 2.2.1 输入模块——读取文件夹图片

**功能描述**：用户指定一个文件夹路径，程序递归读取该文件夹下所有图片文件。

**输入要求**：
- 支持图片格式：JPG、JPEG、PNG、BMP、TIFF
- 支持PDF格式（可选）：文本型PDF和图像型PDF

**交互方式**：
- 图形界面：提供“选择文件夹”按钮，弹出文件夹选择对话框；
- 或命令行参数：通过`--input`参数指定文件夹路径。

#### 2.2.2 图像预处理模块

**功能描述**：对原始发票图片进行预处理，提高OCR识别准确率。

**预处理操作**：
- 灰度化：将彩色图像转为灰度图；
- 二值化：将图像转为黑白，增强文字与背景对比度；
- 去噪：使用高斯模糊等方法去除图像噪声；
- 倾斜校正：检测并矫正发票图像的倾斜角度；
- 分辨率检查：提示用户分辨率低于300dpi的图片可能影响识别效果。

#### 2.2.3 OCR识别模块

**功能描述**：对预处理后的发票图片进行OCR识别，提取文字信息。

**支持的发票类型**：
- 增值税专用发票；
- 增值税普通发票（纸质）；
- 增值税电子普通发票；
- 全电发票（新版全国统一电子发票）。

**识别字段**（至少包含以下核心字段）：

| 字段 | 说明 | 是否必填 |
|------|------|----------|
| 发票代码 | 唯一标识发票所属区域及类型 | 是 |
| 发票号码 | 发票的唯一序列号 | 是 |
| 开票日期 | 发票开具的具体时间 | 是 |
| 合计金额（不含税） | 不含税金额 | 是 |
| 合计税额 | 税额 | 是 |
| 价税合计（含税金额） | 实际支付总金额 | 是 |
| 购买方名称 | 购买方企业名称 | 否 |
| 购买方纳税人识别号 | 购买方税号 | 否 |
| 销售方名称 | 销售方企业名称 | 否 |
| 销售方纳税人识别号 | 销售方税号 | 否 |
| 校验码 | 发票校验码 | 否 |

#### 2.2.4 数据解析与结构化模块

**功能描述**：将OCR识别的原始文本结果解析为结构化的字段数据。

**处理逻辑**：
- 基于正则表达式或关键词匹配提取各字段；
- 对金额字段进行数值格式校验（是否为合法数字）；
- 对日期字段进行格式标准化（统一为YYYY-MM-DD格式）；
- 对纳税人识别号进行长度校验（18位或20位），自动过滤无效字符。

#### 2.2.5 数据校验与异常处理模块

**功能描述**：对识别结果进行自动校验，标记可疑数据。

**校验规则**：
- 必填字段完整性检查（发票代码、号码、金额、日期）；
- 金额数值合法性检查（大于0）；
- 日期格式合法性检查；
- 发票号码重复性检查（同一批次内）。

**异常处理**：
- 识别失败或校验不通过的记录，在Excel中标记为“待人工复核”；
- 生成处理日志，记录每张图片的处理状态（成功/失败/需复核）。

#### 2.2.6 Excel导出模块

**功能描述**：将识别结果导出为结构化的Excel文件，保存到`outputs`目录。

**输出目录结构**：
- 程序运行后，在项目根目录下的`outputs`文件夹中，**以当前时间创建子目录**，格式为`YYYYMMDD_HHMMSS`（例如`outputs/20260617_143022/`）。
- 所有本次运行的输出文件均存放在该时间戳子目录下，确保不同批次结果隔离，便于追溯。

**输出文件清单**（共2个Excel文件）：

1. **模板文件**（`发票报销模板.xlsx`）
   - 存放在时间戳子目录下；
   - 内容为预设的标准报销表头（仅含核心财务字段，不含文件名、状态等辅助信息）；
   - 用户可参考该模板的字段顺序和格式要求，用于后续手工补充或核对。

2. **核心报销数据文件**（`报销明细_YYYYMMDD_HHMMSS.xlsx`）
   - **此文件为最终用于报销提交的正式文件**；
   - **必须剔除所有与报销无关的辅助列**，包括但不限于：“原始文件名”、“识别状态”、“处理时间戳”、“置信度”等；
   - **仅保留以下核心报销字段**（列顺序固定如下）：

| 列序号 | 列名（中文） |
|--------|-------------|
| 1 | 发票代码 |
| 2 | 发票号码 |
| 3 | 开票日期 |
| 4 | 不含税金额 |
| 5 | 税额 |
| 6 | 价税合计 |
| 7 | 购买方名称 |
| 8 | 购买方纳税人识别号 |
| 9 | 销售方名称 |
| 10 | 销售方纳税人识别号 |

- **格式要求**：Excel文件格式为`.xlsx`，编码UTF-8 with BOM（确保Windows Excel打开不乱码），金额保留2位小数，日期格式为`YYYY-MM-DD`。

**辅助日志文件（可选）**：
- 为便于开发调试和问题排查，可在同一子目录下额外输出一份`处理日志.xlsx`或`processing.log`，该日志可包含“原始文件名”、“识别状态”、“错误信息”等辅助信息，但**此日志文件不得与上述核心报销数据文件合并，也不得影响核心文件的纯净性**。


## 三、非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|------|------|
| 单张发票识别时间 | ≤ 3秒 |
| 批量处理能力 | 支持单次处理≥100张图片 |
| 识别准确率 | 清晰图片关键字段≥95% |

### 3.2 可用性需求

- 操作简单，用户只需选择文件夹即可完成全部流程；
- 提供清晰的处理进度反馈；
- 错误信息直观易懂，便于用户定位问题。

### 3.3 数据安全需求

- 所有识别过程在本地完成，发票图片和数据不上传任何外部服务器；
- 不存储用户发票图片的副本（临时文件处理完毕后自动清理）。

### 3.4 兼容性需求

- 操作系统：Windows 10/11（优先）、macOS、Linux；
- Python版本：3.7及以上；
- 如打包为exe，无需安装Python环境即可运行。


## 四、开发环境与流程规范

### 4.1 环境管理（Conda）

- **必须使用Conda管理Python虚拟环境**，严禁在系统基础Python环境中进行开发。
- **环境名称（推荐）**：`invoice-reimburse-ocr`（含义：发票-报销-OCR识别，见名知意）。
- **创建命令**：
  ```bash
  conda create -n invoice-reimburse-ocr python=3.9 -y
  ```
- **激活命令**（开发及运行期间均需保持激活状态）：
  ```bash
  conda activate invoice-reimburse-ocr
  ```
- **依赖安装**：所有第三方库（paddleocr、opencv、pandas等）均须在该激活环境下使用 `pip install` 或 `conda install` 安装。
- **环境导出**（便于他人复现）：
  ```bash
  conda env export > environment.yml
  ```

### 4.2 开发与运行约束

- 所有代码编写、调试运行、测试执行，均须在上述Conda环境激活状态下完成。
- 项目根目录下须同时提供 `requirements.txt`（pip依赖）和 `environment.yml`（完整Conda环境），以便不同场景下复现环境。
- 推荐在IDE（如VS Code、PyCharm）中显式指定 `invoice-reimburse-ocr` 作为项目解释器。


## 五、技术选型建议

### 5.1 OCR引擎选择

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **PaddleOCR** | 免费开源，中文识别优秀，内置发票模型 | 需安装PaddlePaddle依赖 | **推荐**：本地离线部署 |
| Tesseract OCR | 开源，支持100+语言 | 发票模板需定制训练 | 预算有限的轻量场景 |
| 商业OCR API（阿里云/腾讯云/百度云） | 识别准确率高（99%+） | 按调用次数收费，需联网 | 对准确率要求极高 |

**推荐方案**：PaddleOCR（本地免费，中文发票场景表现优秀）。

### 5.2 核心技术库

| 用途 | 推荐库 | 说明 |
|------|--------|------|
| OCR识别 | `paddleocr` | 百度开源OCR工具 |
| 图像预处理 | `opencv-python` | 图像处理 |
| 图像基础操作 | `Pillow` | 基础图像处理 |
| Excel导出 | `pandas` + `openpyxl` | 数据处理与Excel读写 |
| PDF处理（可选） | `pdf2image` | PDF转图片 |
| 打包（可选） | `PyInstaller` | 打包为独立exe |

### 5.3 环境配置文件示例（requirements.txt）

```txt
paddlepaddle>=2.5.0
paddleocr>=2.7.0
opencv-python>=4.8.0
Pillow>=10.0.0

openpyxl>=3.1.0
```

（注：paddlepaddle需根据CPU/GPU版本选择具体安装指令，建议在README中说明）

### 6.2 各模块职责与函数接口约定

| 模块文件 | 核心函数/类 | 输入 | 输出 | 职责说明 |
|----------|-------------|------|------|----------|
| `input_loader.py` | `scan_images(folder_path)` | 用户指定的文件夹路径 | 图片文件路径列表（`List[str]`） | 递归遍历文件夹，筛选出 `.jpg/.jpeg/.png/.bmp/.tiff` 格式文件，支持PDF（可选） |
| `preprocessor.py` | `preprocess(image_path)` | 单张图片路径 | 预处理后的图像对象（`np.ndarray`）或临时文件路径 | 执行灰度化、二值化、去噪、倾斜校正，返回可供OCR引擎直接读取的格式 |
| `ocr_engine.py` | `extract_text(image)` | 预处理后的图像对象 | OCR原始识别结果（`str` 或 `dict`，含文本及置信度） | 调用PaddleOCR，返回全量识别文本及每个字段的置信度 |
| `data_parser.py` | `parse_ocr_result(raw_text)` | OCR原始文本字符串 | 结构化数据字典（`dict`，包含10个核心字段） | 使用正则表达式提取10个报销字段；执行必填项、金额正数、日期格式校验；校验失败字段置为`None`并记录原因 |
| `excel_exporter.py` | `export_to_excel(data_list, output_root)` | 结构化数据列表（`List[dict]`） + 输出根目录 | 无（直接生成文件） | 1. 按当前时间创建子目录；2. 生成仅含表头的模板文件；3. 过滤掉非报销列，生成纯净的核心报销数据Excel |
| `main.py` | `main()` | 命令行参数（`--input`） | 无（控制台输出日志及进度） | 串联整个流程：调用扫描→逐张预处理→OCR→解析→汇总→导出；打印处理进度及最终统计（成功/失败/总数） |

### 6.3 数据流转约定

```text
用户指定文件夹
       ↓
[input_loader] → 图片路径列表
       ↓
遍历每张图片：
  [preprocessor] → 增强图像
       ↓
  [ocr_engine] → 原始文本
       ↓
  [data_parser] → 结构化字典（含状态标记）
       ↓
汇总所有记录 → List[dict]
       ↓
[excel_exporter] → outputs/时间戳/ 目录下生成2个Excel文件
```


## 七、测试要求

### 7.1 自测试脚本规范

- **目录结构**：在项目根目录下创建`tests`文件夹。
- **脚本文件**：在`tests`目录下编写自测试脚本，建议命名为`test_invoice_ocr.py`，或使用`pytest`框架组织多个测试文件。
- **执行方式**：在激活的Conda环境（`invoice-reimburse-ocr`）下，运行：
  ```bash
  # 方式一：直接运行
  python tests/test_invoice_ocr.py
  # 方式二：使用pytest（如安装）
  pytest tests/
  ```

### 7.2 测试覆盖范围

自测试脚本须覆盖以下核心功能点，确保软件符合全部需求预期：

| 测试编号 | 测试项 | 验证内容 | 通过标准 |
|----------|--------|----------|----------|
| TC-01 | 文件夹扫描 | 能否正确递归读取指定文件夹下的所有jpg/png/bmp/tiff图片 | 扫描到的文件数量与实际图片数量一致 |
| TC-02 | 图像预处理 | 预处理流程（灰度、二值化、倾斜校正）是否正常运行不报错 | 所有测试图片预处理通过率100% |
| TC-03 | OCR识别与字段提取 | 使用预先准备的测试发票样本（至少3种不同类型），验证关键字段是否被正确提取 | 必填字段提取准确率≥90%（测试样本） |
| TC-04 | 输出目录结构 | 是否在outputs下创建了`YYYYMMDD_HHMMSS`格式的子目录 | 子目录名称符合时间戳格式 |
| TC-05 | 模板文件生成 | 子目录下是否生成了`发票报销模板.xlsx`，且表头为纯财务字段 | 文件存在且表头不含文件名/状态列 |
| TC-06 | 核心报销Excel | 是否生成了`报销明细_时间戳.xlsx`，且**不包含**“文件名”、“识别状态”等非报销列 | 列名列表与需求2.2.6规定的10个核心字段完全一致 |
| TC-07 | 异常处理 | 传入非图片文件或空文件夹时，程序是否优雅报错而不崩溃 | 程序输出明确错误提示，退出码正常 |

### 7.3 测试数据准备

- 在`tests/`目录下创建`fixtures/`子目录，存放用于测试的样本发票图片（至少包含：增值税专用发票、增值税普通发票、全电发票各一张）。
- 样本图片可脱敏处理（遮盖税号、金额等敏感信息），但需保留关键字段位置以便验证识别逻辑。
- 测试脚本应输出明确的`PASS`/`FAIL`结果，并生成测试日志。


## 八、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 模糊/低质量图片识别率低 | 识别结果不准确 | 增加图像预处理；在日志中记录低置信度样本 |
